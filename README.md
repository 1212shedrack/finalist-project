# Amaranthus Disease Detection and Recommendation System

A professional AI-powered web application that detects diseases in Amaranthus plants from leaf photographs and delivers multilingual treatment recommendations. The system is built with Django, powered by a TensorFlow Lite EfficientNet model, and deployed on Render with full Progressive Web App (PWA) support.

**Live Application:** https://final1v.onrender.com

## Project Overview

The Amaranthus Disease Detection and Recommendation System is an agricultural AI platform developed to assist farmers, agronomists, and researchers in identifying four conditions of the Amaranthus plant: healthy, leaf spot, white rust, and non-Amaranthus. A user uploads a photograph of a leaf, the system runs inference using a TensorFlow Lite model, and then returns a confidence score, risk level, and a full recommendation report covering possible causes, treatment steps, preventive measures, and suggested pesticides. All recommendation content is fully translated into English, Swahili, and French.

## Configuration Files

### `manage.py`
The Django command-line utility entry point. Used locally to run the development server, apply database migrations, collect static files, and compile translation messages. It is not used in production.

### `requirements.txt`
Declares all Python packages required to run the application in production. It intentionally uses `ai-edge-litert` instead of the full `tensorflow` package to keep the Render build fast and within memory limits. The nine packages listed cover the framework, server, static file serving, machine learning runtime, image processing, and translation utilities.

### Procfile
A single-line file that tells Render and similar platforms how to start the application. It launches Gunicorn pointing at the Django WSGI entry point with two worker processes and a 120-second timeout, which accounts for the TFLite model loading time on cold starts.

### `render.yaml`
Render's infrastructure-as-code configuration file. It defines the service type, region, Python version, build command, start command, and all environment variables. The build command runs `pip install`, `collectstatic`, and `migrate` in sequence on every deploy. The `SECRET_KEY` environment variable is auto-generated by Render for security.

### `runtime.txt`
Pins the Python version to `python-3.12.3`. This prevents Render from auto-detecting the project as a Flask application and ensures it reads the `Procfile` correctly for the Django startup command.

### `.gitignore`
Excludes files that must not be committed to GitHub. This includes the virtual environment folder, the generated `staticfiles/` directory, the `db.sqlite3` database, uploaded media files, environment variable files, and IDE configuration folders.

## Application Core

### `amaranthus_project/` (Project Configuration Package)

This folder is the Django project configuration package. It is generated by Django and holds global settings, URL routing, and WSGI/ASGI entry points.

**`settings.py`**
The central configuration file for the entire Django project. It reads `SECRET_KEY` and `DEBUG` from environment variables so the same codebase works in both development and production without modification. Key configurations include WhiteNoise middleware for static file serving, LocaleMiddleware for multilingual support, `ALLOWED_HOSTS` that includes `.onrender.com` for Render deployment, paths to the TFLite model and recommendation JSON files, file upload size limits, and security headers that activate automatically when `DEBUG` is `False`.

**`urls.py`**
The root URL configuration. It connects the project URL namespace to the `disease_app` URL patterns using `i18n_patterns` so language prefixes are handled automatically. It also registers the `/switch-lang/` endpoint outside the language prefix system to allow manual language switching, and mounts the Django admin panel at `/admin/`.

**`wsgi.py`**
The Web Server Gateway Interface entry point. Gunicorn uses this file in production to interface between the web server and the Django application. It sets the `DJANGO_SETTINGS_MODULE` environment variable and exposes the `application` callable that Gunicorn requires.

**`asgi.py`**
The Asynchronous Server Gateway Interface entry point. It is included for future compatibility with async Django features and WebSocket support. The application currently runs in synchronous WSGI mode.

---

### `disease_app/` (Main Application Package)

This is the primary Django application that contains all business logic, database models, views, URL patterns, and the machine learning inference engine.

**`models.py`**
Defines the `Prediction` database model, which stores one record for every leaf analysis performed. Each record contains the uploaded image path, the predicted class key, a human-readable display name, the confidence percentage, probabilities for all four classes stored as a JSON field, the risk level label, the risk colour code used for Bootstrap badge styling, and a timestamp set automatically at creation. This model powers the prediction history page and the admin panel.

**`admin.py`**
Registers the `Prediction` model with the Django admin interface and configures a professional admin view. It provides a list display with all key fields, search by predicted class and display name, filtering by risk level and creation date, a read-only image preview rendered as an HTML `<img>` tag, and a CSV export action that allows administrators to download the full prediction history.

**`forms.py`**
Defines the `ImageUploadForm`, a Django form with a single `ImageField`. The form's `clean_image` method validates that the uploaded file is one of the allowed MIME types (JPEG or PNG) and does not exceed the 10 MB size limit defined in settings. It returns clear error messages if either check fails.

**`views.py`**
Contains all the view functions that handle HTTP requests and return responses. The `home` view renders the landing page and passes recent predictions and site-wide statistics from the database. The `predict` view accepts a POST request with an image, saves it to disk, calls the predictor module, saves the result to the database, and redirects to the result page. The `result` view retrieves a specific prediction by primary key and loads the recommendation content in the user's active language. The `history` view queries all predictions with optional keyword search and paginates the results. The `about` view renders the static informational page. The `delete_prediction` view handles deletion with a CSRF-protected POST request. The `statistics` view returns a JSON object used by the home page to animate counter elements. The `switch_language` view handles language switching by manually setting the language cookie and redirecting back to the same page.

**`predictor.py`**
The machine learning inference engine. It loads the TFLite model once at Django startup using `ai-edge-litert` and falls back to `tensorflow.lite` if `ai-edge-litert` is not available. Image preprocessing is performed with Pillow and NumPy, matching the exact pipeline the model was trained with: open as RGB, resize to 224×224 pixels, convert to a float32 array, and expand dimensions to shape `[1, 224, 224, 3]`. No division by 255 is applied because the model handles normalisation internally. The `predict_image` function runs inference and returns the predicted class, display name, confidence score, all four class probabilities, risk level, and risk colour. The `load_recommendation` function accepts a class name and a language code, then reads from the matching JSON file (`recommendations.json`, `recommendations_sw.json`, or `recommendations_fr.json`), falling back to English if the language file is not found.

**`urls.py`**
Defines the URL patterns for the `disease_app` application under the `disease_app` namespace. The patterns map the home page, image upload page, prediction result page, prediction history, about page, individual prediction deletion, and the statistics JSON endpoint.

**`apps.py`**
The Django application configuration class. It sets the default auto field type and registers the application under the name `disease_app`.

**`migrations/0001_initial.py`**
The initial database migration file generated by Django. It creates the `disease_app_prediction` table in SQLite with all the fields defined in `models.py`. This file is applied automatically during deployment when the build command runs `python manage.py migrate`.

## Templates

All HTML templates extend `base.html` using Django's template inheritance system. Templates use the `{% trans %}` tag throughout so all static text is passed through Django's translation engine.

**`base.html`**
The master layout template that all other pages inherit. It defines the complete HTML document structure including the `<head>` section with PWA meta tags, the Web App Manifest link, Apple Touch Icon references, Bootstrap 5 and Bootstrap Icons CDN links, and the Poppins Google Font. The body contains the responsive navbar with the brand logo, navigation links, a dark mode toggle, and the language switcher. The footer contains links, contact information, and project credits. At the bottom of the body, the PWA install banner is declared with its install and dismiss buttons, followed by the service worker registration script and the install prompt event handling logic.

**`home.html`**
The landing page template. It displays an animated hero section with a gradient background, a headline, subtitle, and two call-to-action buttons. Below the hero is a statistics bar that shows the total number of predictions, healthy detections, and disease detections, animated with a count-up effect on scroll. The page continues with four feature cards, a three-step how-it-works section, and a grid of recent predictions pulled from the database.

**`predict.html`**
The image upload page template. It presents a drag-and-drop upload zone that accepts JPEG and PNG files. When a file is selected, a JavaScript preview renders the image before submission. A loading overlay appears after the form is submitted to indicate that the model is processing the image.

**`result.html`**
The analysis results page template. It displays the uploaded leaf image alongside the predicted class, confidence score, and risk level badge. A confidence progress bar animates to the confidence value. Below that, probability bars show the scores for all four classes. A tabbed section presents the recommendation content including the summary, treatment steps, prevention measures, recommended pesticides, and next steps, all rendered from the JSON recommendation data in the user's active language.

**`history.html`**
The prediction history page template. It includes a search bar for filtering results by class name, filter chips for quick category filtering, and a responsive grid of prediction cards each showing the image thumbnail, disease name, confidence, risk badge, timestamp, and a delete button. Pagination controls appear at the bottom when the number of predictions exceeds ten.

**`about.html`**
The informational page template. It describes the system's purpose, the four disease classes with their characteristics, the EfficientNet model architecture, the dataset used for training, and the accuracy metrics. It also lists the technology stack used in development.

## Static Assets

### `static/css/style.css`
The main stylesheet containing over 1,500 lines of custom CSS. It defines a set of CSS custom properties for the colour palette, typography, and spacing. Styled components include the dark-green glassmorphism navbar, the animated gradient hero section, hover-lift feature cards, the drag-and-drop upload zone with glow effects on drag, result page confidence bars with animated fills, colour-coded disease badges, the multi-tab recommendation panel, the prediction history grid, the footer, and the PWA install banner with its slide-up animation. A full dark mode theme is implemented using the `[data-theme="dark"]` selector and toggled by JavaScript.

### `static/js/main.js`
The main client-side JavaScript file. It handles drag-and-drop file upload with visual feedback, file preview using the FileReader API, client-side file type and size validation before form submission, a loading overlay during form POST, animated progress bars on the result page, a toast notification system, a counter animation for the statistics bar triggered on scroll using IntersectionObserver, a navbar background change on scroll, smooth scrolling for anchor links, dark mode toggle with persistence in localStorage, a delete confirmation dialog for history entries, and debounced search input to reduce unnecessary requests.

### `static/js/service-worker.js`
The PWA service worker script. It uses a Cache First strategy for all static assets so they load instantly after the first visit. It uses a Network First strategy for HTML pages so content stays fresh but falls back to the cache when the user is offline. POST requests to the prediction endpoint are never cached. On install, the service worker pre-caches the home, predict, history, and about pages along with the CSS, JavaScript, and icon files. If the user navigates to a page that is not cached and has no internet connection, the service worker returns a self-contained offline fallback page rendered directly from the script.

### `static/manifest.json`
The Web App Manifest file required for PWA installability. It declares the full app name, short name, description, theme colour, background colour, display mode as standalone, and the two required icon sizes at 192×192 and 512×512 pixels. It also defines two app shortcuts that appear when the user long-presses the home screen icon, allowing direct navigation to the Detect Disease page and the History page.

### `static/favicon.ico`
The browser tab favicon, generated from the app icon at 16×16 and 32×32 pixel sizes.

### `static/icons/`
Contains all generated PNG icon files used for the PWA install icon, Apple Touch Icon, and browser favicons. The sizes provided are 512×512, 192×192, 180×180, 32×32, and 16×16 pixels.

## Recommendation Data

### `recommendations.json`
The English language recommendation data file. It is a JSON object with four top-level keys corresponding to the four disease classes. Each key maps to an object containing the disease title, a summary paragraph, risk level, possible causes, a list of recommended actions, prevention measures, treatment steps, suggested pesticide products with dosage information, practical farmer tips, and immediate next steps. This file is the master data source and is also used as the fallback if a translated file is not found.

### `recommendations_sw.json`
The complete Swahili translation of all recommendation content. It follows exactly the same structure as the English file and covers all four disease classes with full translations of every field including titles, summaries, action lists, pesticide names with dosages, and next steps.

### `recommendations_fr.json`
The complete French translation of all recommendation content. It follows the same structure as the English file and provides accurate agronomic terminology in French for all four disease classes.

The correct file is selected at runtime by the `load_recommendation` function in `predictor.py`, which reads the active language code from the Django request object and maps it to the corresponding filename.

## Localisation

### `locale/`
Contains the Django gettext translation files that translate all static UI text in templates and Python code.

**`locale/sw/LC_MESSAGES/`**
Holds `django.po` (the human-editable translation source file) and `django.mo` (the compiled binary used at runtime) for Swahili translations of all navbar labels, page headings, button text, form labels, and messages.

**`locale/fr/LC_MESSAGES/`**
Holds the equivalent `.po` and `.mo` files for French UI translations.

Language switching is handled by the `/switch-lang/` view, which reads the selected language code from a POST form, sets the Django language cookie, and redirects the user back to the current page. The `LocaleMiddleware` reads this cookie on subsequent requests and activates the appropriate language for template rendering and recommendation loading.

## Machine Learning Model

### `model/amaranthus_efficientnet_v5.tflite`
The TensorFlow Lite model file used for disease classification. The model is based on the EfficientNet architecture, fine-tuned on a dataset of Amaranthus leaf images across four classes. The input tensor expects a float32 array of shape `[1, 224, 224, 3]` with raw pixel values in the range 0 to 255. The model applies internal normalisation so no division by 255 is performed during preprocessing. The output tensor is a float32 array of shape `[1, 4]` representing the softmax probabilities for the four classes in alphabetical order: healthy, non-Amaranthus, leaf spot, and white rust. The model file is 17 MB and is committed directly to the GitHub repository so it is available to Render during the build.

## Deployment

### Render Platform

The application is hosted on Render's free tier at **https://final1v.onrender.com**. Every push to the `main` branch on GitHub triggers an automatic redeploy. The build process installs Python packages, runs `collectstatic` to compile all static files into the `staticfiles/` directory, and applies any pending database migrations. WhiteNoise serves the compiled static files directly from the Django process without requiring a separate web server like Nginx.

The `SECRET_KEY` is auto-generated and stored securely as a Render environment variable. `DEBUG` is set to `False` in production, which enables security headers and disables the Django debug page. The SQLite database is stored on Render's ephemeral file system, which means prediction history is reset on each redeploy. This is acceptable for a research and demonstration deployment.

### Progressive Web App

The application meets all PWA requirements: it is served over HTTPS, has a valid Web App Manifest, and registers a service worker. On supported browsers, users are prompted to install the application to their home screen or desktop. Once installed, the app opens in standalone mode without a browser address bar, behaves like a native application, and shows a branded offline page when there is no internet connection.

## Installation and Local Setup

**Prerequisites:** Python 3.12, Git

```bash
# Clone the repository
git clone https://github.com/1212shedrack/finalist-project.git
cd finalist-project

# Create and activate a virtual environment
python -m venv venv
venv\Scripts\activate          # Windows
source venv/bin/activate       # macOS / Linux

# Install dependencies
pip install -r requirements.txt

# Apply database migrations
python manage.py migrate

# Collect static files
python manage.py collectstatic --no-input

# Start the development server
python manage.py runserver
```

Open `http://127.0.0.1:8000` in your browser.

## Disease Classes

| Class Key | Display Name | Risk Level | Description |
|---|---|---|---|
| `health` | Healthy | Low | No visible disease. Plant shows normal colour and structure. |
| `spot_leaf` | Leaf Spot | Medium | Fungal infection causing dark lesions. Caused by Alternaria or Cercospora species. |
| `white_rust` | White Rust | High | Oomycete infection caused by Albugo bliti. Appears as chalky white pustules. Spreads rapidly. |
| `non_amaranthus` | Non-Amaranthus | Unknown | The uploaded image was not identified as an Amaranthus leaf. |
